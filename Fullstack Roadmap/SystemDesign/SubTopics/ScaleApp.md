The video outlines a step-by-step approach to scaling a web application from handling a single user to supporting millions, emphasizing system design principles for performance, reliability, and efficiency. It starts with the simplest setup and progressively introduces components to manage growing traffic and data.
The initial architecture involves a user device connecting via DNS to a single web server. This server manages everything: the user interface, business logic, and data storage in an integrated database. All requests—reads and writes—are handled by this one machine. As user numbers increase and data accumulates, the system faces bottlenecks, such as the server becoming overwhelmed or storage filling up.
To address this, the database is decoupled from the web server into a dedicated database server. This separation allows the web server to focus on processing requests while the database handles persistence. The web server communicates with the database over a network, improving modularity but still limiting the system to a single point of failure.
As traffic grows further, vertical scaling is considered, which means upgrading the single server's hardware by adding more CPUs, RAM, or storage. However, this has limits due to physical constraints and cost, and it doesn't provide redundancy—if the server fails, the entire system goes down.
The preferred solution is horizontal scaling, where multiple identical web servers are added to form a pool. This distributes the load and provides fault tolerance: if one server fails, others can take over. Requests are routed through a load balancer, which intelligently directs traffic to the least busy server based on factors like current load or response time. The load balancer also masks internal server IPs from users, enhancing security by preventing direct access. Publicly, users interact only with the load balancer's IP.
For the database layer, a single database is insufficient for high traffic. Database replication is introduced, creating a master-slave setup. The master database handles all write operations (inserts, updates, deletes), while multiple slave databases are read-only replicas that sync data from the master and handle read operations. This separation leverages the fact that reads often outnumber writes in web apps, allowing reads to scale by adding more slaves. A database load balancer or routing logic in the application directs writes to the master and reads to slaves.
In case of failures:

If a slave fails, reads are redirected to other slaves or temporarily to the master until a replacement slave is set up and synced.
If the master fails, a slave is promoted to become the new master. The process involves selecting a slave with the most current data, elevating it to master status, redirecting writes to it, running data recovery scripts to reconcile any missing transactions from the failed master (if recoverable), and then attaching new slaves to the new master for continued replication. The video notes that replication isn't always real-time, so the promoted slave might miss recent writes, requiring synchronization to avoid data inconsistencies. No specific algorithm names are mentioned for this promotion process; it's described as a complex operation often handled manually or with scripts in basic setups, with advanced alternatives like multi-master or circular replication referenced but not detailed as they add complexity and are beyond the basic scope.

To further optimize response times, a cache is inserted between the web servers and databases. The cache stores frequently accessed data as key-value pairs (e.g., using systems like Redis or Memcached). When a server needs data, it first checks the cache; if present (a cache hit), it's served quickly. If not (a cache miss), the database is queried, and the result is stored in the cache for future requests. Key considerations include eviction policies (to remove old data when the cache fills), expiration times (to prevent stale data), and consistency mechanisms (to update the cache when underlying data changes). This reduces database load significantly for read-heavy workloads.
For global users experiencing latency due to distance from servers, a Content Delivery Network (CDN) is added. CDNs consist of geographically distributed servers that cache static content like images, videos, CSS, JavaScript, and HTML. User requests are routed to the nearest CDN edge server via DNS. If the content isn't cached locally, the CDN fetches it from the origin servers, stores it with a Time to Live (TTL) value to control freshness, and serves it. This minimizes data travel time, improving load speeds especially for media-rich apps.
Session management becomes an issue in a multi-server setup, as users might be routed to different servers mid-session. To handle this, session data is moved to shared storage, creating a stateless architecture where any server can process any request without local state. Shared storage options include relational databases, caches, or NoSQL databases, with NoSQL preferred for its scalability in handling unstructured or semi-structured data.
To monitor and log system performance asynchronously, message queues are implemented. Web servers push non-critical events (like errors, resource usage, or analytics) into queues, where worker processes consume and handle them separately. This prevents blocking the main request-handling flow, ensuring responsiveness.
Finally, for true global scale and disaster recovery, the system is replicated across multiple data centers in different regions. Traffic is directed to the closest data center, and if one fails (e.g., due to outages or disasters), DNS reroutes to another. This multi-region setup combines all prior elements—load balancers, replication, caches, CDNs, queues, and stateless servers—across geographies for maximum availability.
Overall, the architecture evolves by layering horizontal scaling, load balancing, replication for redundancy, caching for speed, CDNs for low latency, shared storage for statelessness, queues for async processing, and multi-data-center deployment for resilience, enabling the system to handle millions of users reliably.
