# Microfrontends (MFEs) in React – Architecture Guide

## 1. What are Microfrontends?

Microfrontends (MFEs) are an architectural pattern where a large frontend application is split into **smaller, independently deployable applications**.

Instead of one large monolithic React app:

```
Monolith App
 ├─ Dashboard
 ├─ Payments
 ├─ Profile
 └─ Settings
```

We split it into multiple applications:

```
Host Application
 ├─ Dashboard MFE
 ├─ Payments MFE
 ├─ Profile MFE
 └─ Settings MFE
```

Each MFE:

* Has its **own repository**
* Has **independent CI/CD**
* Can be **deployed separately**
* Can even use **different frameworks**

---

# 2. Why Microfrontends are Used

### Problems with Monolithic Frontends

* Large bundle size
* Long CI/CD pipelines
* Multiple teams working in the same repository
* Difficult scaling of development

Example: Banking platform

Teams:

* Accounts team
* Payments team
* Cards team
* Loans team

Without MFEs → All teams work in **one repository**.

With MFEs → Each team owns its **own frontend module**.

```
accounts.bank.com
payments.bank.com
cards.bank.com
```

A **Host application composes them together**.

# 4. Common Integration Approaches

| Approach               | Description                      |
| ---------------------- | -------------------------------- |
| Build-time integration | MFEs shared via npm packages     |
| Runtime integration    | Webpack Module Federation        |
| Iframes                | Apps loaded in iframe containers |
| Web Components         | Framework-agnostic components    |

Most modern React architectures use **Webpack Module Federation**.

---

# 5. Module Federation

Module Federation (Webpack 5) allows **loading JavaScript modules from another application at runtime**.

This means:

```
Host App → loads code from Remote App
```

Without redeploying the host.

---

# 6. Important Terminology

## Host Application

The **main container application**.

Responsibilities:

* Routing
* Layout
* Loading MFEs

Example:

```
main-shell
```

---

## Remote Application

An independently deployed **microfrontend**.

Example:

```
payments-mfe
```

---

## remoteEntry.js

This is the **entry file generated by Module Federation**.

Example:

```
https://payments.app.com/remoteEntry.js
```

This file contains:

* Module registry
* Exposed components
* Dependency sharing logic

Example runtime structure:

```
window.payments = {
  get: () => {...},
  init: () => {...}
}
```

The **host loads this file first** before importing modules.

---

# 7. MFE Registry / Entry Point

Large systems use a **registry service** that stores all MFE URLs.

Example registry:

```
mfe-registry.json
```

Example:

```json
{
  "payments": "https://payments.app.com/remoteEntry.js",
  "accounts": "https://accounts.app.com/remoteEntry.js",
  "cards": "https://cards.app.com/remoteEntry.js"
}
```

Host loads this dynamically:

```
fetch('/mfe-registry')
```

Benefits:

* Runtime discovery of MFEs
* Version control
* Canary releases
* Feature flag routing

# 8. Remote MFE Configuration

Example: **payments-mfe**

### webpack.config.js

```javascript
const ModuleFederationPlugin =
  require("webpack").container.ModuleFederationPlugin;

module.exports = {
  plugins: [
    new ModuleFederationPlugin({
      name: "payments",

      filename: "remoteEntry.js",

      exposes: {
        "./PaymentsApp": "./src/bootstrap",
      },

      shared: {
        react: { singleton: true },
        "react-dom": { singleton: true }
      }
    })
  ]
};
```

Explanation:

| Property | Meaning                         |
| -------- | ------------------------------- |
| name     | global container name           |
| filename | generated federation entry file |
| exposes  | modules exposed to host         |
| shared   | shared dependencies             |

```javascript
    react: { singleton: true },
    "react-dom": { singleton: true }
  }

```

# 12. Dynamic Runtime Loading

Instead of hardcoding URLs, MFEs can be loaded dynamically.

Example loader:

```javascript
function loadRemote(url, scope) {
  return new Promise((resolve) => {
    const script = document.createElement("script");

    script.src = url;

    script.onload = () => {
      resolve(window[scope]);
    };

    document.head.appendChild(script);
  });
}
```

Example usage:

```
loadRemote("https://payments.app.com/remoteEntry.js","payments")
```

# 14. Shared Dependencies

Without sharing:

```
Host → React
Remote → React
```

Two React instances get bundled.

With sharing:

```javascript
shared: {
  react: { singleton: true },
  "react-dom": { singleton: true }
}
```

This ensures **only one instance of React exists**.

---

# 15. What Happens if Multiple React Instances are Loaded

If MFEs load their own React versions, several issues occur.

### Duplicate React Runtime

Example:

```
Host → React 19
Remote → React 18
```

This causes **two React trees running in the same page**.

---

### Hook Failures

React hooks depend on a **single renderer instance**.

Example error:

```
Invalid hook call
Hooks can only be called inside the body of a function component
```

---

### Context Breaks

React context will not propagate across different React instances.

Example:

```
Host ThemeProvider
      ↓
Remote component
```

Remote will **not receive the context**.

---

### State Inconsistencies

React state updates may not propagate correctly across MFEs.

Example:

```
Host Redux store
Remote uses another React instance
```

Provider connections break.

---

### Increased Bundle Size

Multiple React versions increase bundle size significantly.

Example:

```
React (120KB)
ReactDOM (120KB)

Two versions → 240KB+
```

---

# 16. How to Prevent Multiple React Instances

Use **shared singleton dependencies**.

Example:

```javascript
shared: {
  react: {
    singleton: true,
    requiredVersion: "^19.0.0"
  },
  "react-dom": {
    singleton: true
  }
}
```

This ensures:

* One React instance
* Dependency deduplication
* Smaller bundles

---

# 17. Deployment Model

Each MFE deploys independently.

Example pipeline:

```
Payments MFE
   ↓
Build
   ↓
Upload remoteEntry.js to CDN
   ↓
Host loads automatically
```

Host does **not need redeployment**.

# 19. Performance Considerations

MFEs introduce additional network calls:

```
remoteEntry.js
chunk1.js
chunk2.js
```

Optimizations:

* CDN caching
* HTTP/2 multiplexing
* code splitting
* prefetching remoteEntry

---

# 20. When NOT to Use Microfrontends

Avoid MFEs when:

* The application is small
* Team size is small
* Simple UI requirements

MFEs add architectural complexity.

# 21. Summary

Microfrontends enable **scalable frontend architecture**.

Core components:

```
Host application
Remote MFEs
remoteEntry.js
Module Federation
Shared dependencies
MFE registry
```

Key benefits:

* Independent deployments
* Team scalability
* Runtime module loading

Key risks:

* Dependency conflicts
* Multiple framework instances
* Runtime integration complexity
