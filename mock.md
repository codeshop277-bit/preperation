# Question 1
How Module Federation works under the hood, and how you handled shared dependencies and version conflicts?

Module federation allows integrating independent applications during the runtime. Each remote repo exposes a remote entry file, which acts as a container exposing module factories. The host repo consumes the exposed containers and it loads the remote repo dynamically. So once the remote is loaded dynamically, the shared dependencies initialization step begins. This is where Webpack negotiates between the host and the remote versions of the dependencies. So whatever the dependencies are initialized in the shared part, will be reused from the parent and whatever which are not initialized, will be loaded from the remote. And Webpack resolves the dependency version issue by using the required version or stricter version attributes. With this we can enforce a certain version of the dependency to be loaded. So if it is not available, it falls back to the closest available version. And for some of the dependencies, we need to ensure to add a singleton to group. This will make sure only one instance of that dependency is available in both the host and remote parts. This prevents the duplicated bundle issue for the same dependency. For example, React must be declared as a singleton instance because hook call relies on the single instance. in our project, we faced this invalid hooks calls due to the React duplicate dependency issue. We finally sorted it out by imposing singletons of the React dependencies. While module federation allows team independence, it introduces a runtime dependency and also version management complexity. That is the trade-off that needs to be considered as well.

# Question 3:

In React (especially with large applications like yours),

👉 How do you prevent unnecessary re-renders?
So re-render mostly happens due to the state change or a props change. So we can make use of the React hooks to prevent this. We can start off with the useMemo. For example, if we are doing a filter operation on large sets of data, so we can use useMemo for that operation. What it does is only if there is a change in the props, this will redo the filter option again. Or else, it will return the computed output. And we can also memoize the whole component. We can use React.memo to memoize the whole component. This ensures the component will re-render only when there is a change in the props and it will skip it if there is no change in props. And the other hook is useCallback. So when a function is passed as a callback to a child component, when the parent re-renders, the child will also re-render due to a change in state. Even though the function itself is not changed, the re-render happens due to a change in the reference of the function. To prevent this type of re-renders, what we can do is we can wrap the child component in the React.memo and we can wrap the function in the parent component using a useCallback hook. useCallback hook will handle the change in reference and child component will not get re-rendered since it has been memoized and it passes the shallow check. Apart from this, we can also make use of the React Profiler tool, so it gives us the list of the components which are getting re-rendered on any state changer. We can also follow state colocation where a state of the component is handled closely to its existence which prevents multiple renders. And while using context make sure to split the context else it would cause renrenders across app. And for rendering a virtually large list, for example, 5000 rows, what we can do is we can use the React window to limit the number of rows that are being rendered in the viewport, and we can memoize the filter operations and we can also use React.memo for rendering the row components, so it will not get re-rendered when there is not much props change. And using useMemo and React.memo also comes with an increasing memory cost because for every memoized output, it will be stored in the memory, so that is one trade-off you should also consider while using memo.

# Question 5:

👉 In a large React application, how do you decide when to use Context vs Redux (or any global state library)?


useContext and Redux, both are used for state management purposes. useContext is more ideal when we have a small application or when the state updates are minimal. So what it does is when we create a context, so whatever the child components that are wrapped inside the context will get rerendered when there is a state change. So when the app size grows and we have multiple contexts created, this would become an issue since maintaining the wrapping order would become messy and this would also rerender the entire application. Redux is more ideal for when there are multiple state changes or more frequent state changes and multiple components are reused the state from the centralized store. And Redux also comes up with the DevTools feature by which we can able to debug what state, what are the payloads are being sent to the store and how it is being updated there.And with the Redux toolkit, Redux, we can also make use of the RTK query, which would be useful for, you know, the calling the APIs, and it also comes with the caching and deduplication methods, which would help in the effective API calls. Context can also be used, but it is not ideal for using it. Use context is also used for the client state, for example, the theme context or the auth contextAnd Redux is more used for the server state for managing the API data and the application's internal state when there are more updates. and for example for a real world e-commerce site we can have multiple slices created for each functionality like auth, orders, products, payments and maintain its own state. useContext can be used directly. It has no configuration or any steps, but as the app grows, maintaining multiple contexts would become messy and it would be prone to multiple re-renders. Whereas Redux comes up with the initial configuration cost. By cost, I mean the initial work which we need to do, the initial extra effort we need to put in while doing the configuration for the Redux store. But that would be the price we would be paying for the more seamless state management. So even when the app size grows, it would not introduce any issues. It would be a seamless approach.The one more drawback in useContext is it re-renders all the child components inside it, whereas Redux re-renders only the subscribers' selective subscription via the selectors used.
